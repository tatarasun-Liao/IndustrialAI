·第一层：为什么要搞这些？（多核 CPU 的“大忽悠”）
现代计算机通常有多个 CPU 核心（比如 8 核）。每个核心都有自己独立的高速缓存（L1/L2 Cache）。
当两个核心同时操作同一个变量时，它们并不是直接读写内存，而是先读到自己核心的缓存里。

危险场景：
核心 A 把变量 isRunning 改成了 false（写到了自己的缓存里）。
核心 B 因为不知道 A 改了，依然在自己的缓存里读 isRunning（读出来的还是 true）。
结果：核心 B 陷入了死循环，永远不知道程序应该停止。

为了解决这个“缓存不一致”和“代码乱序执行”的问题，我们需要“封印”CPU 的投机取巧，这就引出了三个兵器：内存屏障、volatile、Interlocked。

·第二层：内存屏障（Memory Barrier）—— 硬核的“防护栏”
内存屏障本质是一条特殊的 CPU 指令。它像一道物理栅栏，强行限制 CPU 对读写指令的“乱序执行”和“缓存延迟”。

读屏障（Acquire）：在读取指令之前，必须把 CPU 缓存里所有“未落盘（未写入主存）”的读取操作全部清空/刷新，才能执行后面的读操作。

写屏障（Release）：在写入指令之后，保证之前的所有写入操作已经全部刷新到主存中，才允许执行后续的操作。

在 C# 中，最底层的原生写法是 Thread.MemoryBarrier()。
// 假设有两个线程，如果不用 MemoryBarrier，CPU 可能会把下面两行顺序颠倒
var a = 1;
Thread.MemoryBarrier(); // 强制屏障！确保 a=1 写入完成后，才允许执行 b=2
var b = 2;
在 C# 业务代码中，我们几乎不会直接写 Thread.MemoryBarrier()。因为 volatile 和 Interlocked 在底层已经默默帮我们插入好了这些屏障，我们只需要用高级版本即可

·第三层：volatile 关键字 —— 轻量级的“防缓存报信员”
volatile 是 C# 中修饰字段的关键字。

volatile 到底干了什么？

防止缓存：告诉 JIT 编译器，这个变量绝对不许被优化缓存到 CPU 寄存器里，每一次读取都必须强行从内存中“亲自”去取最新值。

禁止重排序：底层会给读写这个变量自动加上读屏障（Acquire）和写屏障（Release）。

public class Worker
{
    // 如果不加 volatile，另一个核心可能永远读不到 CPU 缓存里更新的最新值
    private volatile bool _isStopped = false; 

    public void Stop() => _isStopped = true;

    public void Run()
    {
        while (!_isStopped) 
        {
            // 做繁重的业务工作
        }
    }
}

致命大坑！volatile 不能保证原子性！
这是无数新手栽跟头的地方。如果你写：

private volatile int _count = 0;

// 两个线程同时执行下面这行
_count++; 
结果：依然会丢失数据，导致最终 _count 比实际要小！因为 _count++ 在底层被拆分为 3 步：1.读取值 -> 2.加 1 -> 3.写入新值。
volatile 只能保证第 1 步读取的永远是最新值，但无法保证在“第1步”和“第3步”之间，另一个线程没插进来。

·第四层：Interlocked 类 —— 原子的“钢铁手术刀”
如果你需要做加减、比较并交换等操作，且必须保证绝对线程安全，必须使用 Interlocked。

它的原理：它直接调用底层 CPU 的 CAS（Compare And Swap，比较并交换） 或 Lock 前缀 指令。这个指令在硬件层面就是一个完整的 “读取 -> 修改 -> 写入” 原子操作。

Interlocked 可以做到：

1.原子自增/自减（完美解决 _count++ 问题）：

private int _counter;
// 不管多少线程并发执行，最终结果绝对精确！
Interlocked.Increment(ref _counter); 
Interlocked.Decrement(ref _counter);

2.原子交换：

int newValue = 100;
// 把 _value 瞬间替换成 100，并返回原来的值
int oldValue = Interlocked.Exchange(ref _value, newValue);

3.原子条件更新（最核心的 CAS 操作）：

int original = 5;
int newValue = 10;
// 只有当 _value 当前等于 original(5) 时，才将其更新为 newValue(10)。
// 如果中间被别人改了，返回 false，什么也不做。
bool success = Interlocked.CompareExchange(ref _value, newValue, original);
这个 CompareExchange 是构建所有“无锁数据结构”（Lock-Free Data Structure，比如我们之前聊到的 ConcurrentDictionary）的底层灵魂。
