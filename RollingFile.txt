第一层：操作系统的“硬核真相”—— Move 与句柄（Handle）的纠缠
首先，我们要弄懂文件系统底层的两个原子性：

Move / Rename 是原子的：在 Windows 和 Linux 上，重命名文件是一个操作系统内核级别的原子操作。如果在重命名瞬间，另一个线程试图读取该文件，要么读到旧名字，要么读到新名字，不会出现“半半拉拉”的情况。

句柄（Handle）是独立的：这是最核心的陷阱！ 当进程 A 用 FileStream 打开了 app.log，进程 B（滚动线程）把 app.log 重命名为 app.2026.log。

底层发生了什么？ 操作系统只是删除了“app.log”这个文件名，把底层数据节点（Inode）指向了“app.2026.log”。

进程 A 会崩溃吗？ 绝对不会！ 进程 A 手里还紧握着那个文件句柄。它依然能往这个底层的物理节点写入数据。直到进程 A 关闭（Dispose）了文件流，这个物理文件才会真正在磁盘上被释放。

灾难后果：如果滚动发生后，进程 A 还在疯狂写入，数据会全部落在 app.2026.log 里，而你的新日志轮询逻辑却在找新建的 app.log，导致数据彻底“人间蒸发”！


第二层：文件滚动的 4 种“原子性破裂”灾难现场
1. 数据错位灾难（句柄残留）
场景：写线程刚写完一条日志（还在缓冲区），滚动线程直接把文件 Move 走了。写线程毫不知情，继续往旧句柄里写。
结局：数据被写入旧文件，业务上以为日志在按时间分片，实际上最新日志全跑到了几天前的旧压缩包里，排查极其困难。

2. 检查与执行的时间差漏洞（TOCTOU）
场景：滚动线程判断“当前文件大小 > 10MB”，准备滚动。它先检查是否存在 app.1.log，不存在，于是准备执行 File.Move。
灾难：就在“检查完”和“执行 Move”的这几十微秒内，另一个滚动线程抢先一步创建了 app.1.log（或者外部的监控程序删除了它）。
结局：File.Move 直接抛出 IOException，文件滚动失败，导致系统卡死或持续写满磁盘。

3. 截断与写入的对齐灾难
场景：你是通过“复制当前文件 -> 清空原文件（SetLength(0)）”的方式实现滚动。
灾难：在清空原文件（Truncate）的瞬间，写线程刚好有一行数据写入到了原来文件的位置（比如第 100 字节）。因为截断是瞬间发生的，那行数据会被瞬间切掉一半，导致日志文件末尾出现半个字符的乱码或无法解析。这种数据损坏是不可逆的。

4. 多进程死锁灾难
场景：两个 IIS / 控制台进程同时写同一个 app.log。滚动线程 A 锁定文件准备 Move，滚动线程 B 锁定文件准备 Move。
结局：两个进程互相持有对方需要的文件锁，直接造成死锁（Deadlock），整个应用写入卡死，请求超时。


第三层：工业界如何保证“原子性滚动”？
大厂日志框架（如 NLog、Serilog）为了避免上述灾难，底层采用了“三步走”原子锁协议：

1. 写入者必须让路（ReaderWriterLockSlim）
在内存中维护一把读写锁。

写线程：每次写之前，申请 EnterReadLock()（多线程可并发写）。

滚动线程（调度者）：申请 EnterWriteLock()。这个锁会阻塞所有新进来的写线程，并等待当前正在写的线程执行完毕。

结果：在滚动发生的那一瞬间，整个世界是静止的，没有一行日志在被写入。这保证了日志不被截断，也不会有“半行”数据

2. 安全句柄剥离与原子重命名（赌上 FileShare 的操作）
在获得写锁后，滚动线程执行：

// 1. 关键操作：关闭当前的 FileStream，但允许旧句柄上的缓冲区先刷完。
_currentStream.Flush();
_currentStream.Dispose(); // 彻底释放句柄！

// 2. 执行原子重命名。此时因为没有其他句柄持有它，Move 必定成功。
File.Move("app.log", "app.2026.log");

// 3. 重新以追加模式创建 app.log
_currentStream = new FileStream("app.log", FileMode.CreateNew, ...);
注意：在重命名时，如果目标是 app.1.log 且已存在，必须用 File.Replace 或先删后搬，避免竞态异常。

3. 多进程场景下的“全局互斥锁”（Mutex）
如果有多进程（多个 EXE 写同一个日志文件），内存锁无效。工业界最通用的原子隔离方案是 System.Threading.Mutex（命名互斥体）。

滚动前，申请名为 Global\MyAppLogRollingMutex 的互斥锁。

如果当前没进程在滚，申请成功，执行滚动；如果另一个进程正在滚，本进程直接阻塞等待。这保证了同一时间物理上只有一个进程在操作文件夹。


第四层：.NET 中 FileShare 参数的神秘力量（原子性必须懂这个）
当你 new FileStream 时，有一个 FileShare 参数。这是决定原子性生死的关键：

csharp
// 如果你是这样打开日志文件的：
new FileStream("app.log", FileMode.Append, FileAccess.Write, FileShare.ReadWrite);
底层含义：告诉操作系统，允许其他线程/进程同时读取（Read）和写入（Write）这个文件。
致命隐患：这种模式下，你如果用 File.Move，极大概率会失败，因为 FileShare.ReadWrite 往往意味着别人持有了共享句柄，禁止重命名。在某些严格的 Linux 或容器环境下，Move 会直接返回“设备或资源忙”。

工业级标准开局方式（极尽细致）：

csharp
// 必须加上 FileShare.Delete！这是允许重命名/删除的关键！
var fs = new FileStream(
    "app.log", 
    FileMode.Append, 
    FileAccess.Write, 
    FileShare.ReadWrite | FileShare.Delete // ⚠️ 黑科技：允许外部进行重命名操作！
);
只有允许了 FileShare.Delete，滚动线程才能在其持有句柄的同时，安全地执行 File.Move 而不会引发 IO 异常。


第五层：终极实战高度——如何保证“滚动过程中绝对不丢一行”？
哪怕是上面最完美的处理方式，如果日志写入时是异步的（await fs.WriteAsync），依然可能丢数据。

终极的原子性设计（解耦模式）：
真正的工业级日志框架（如 Seq 或 Serilog 的底层），并不直接在一个文件上做“滚动+写”。
而是采用二阶段提交：

前台线程只负责把日志排队塞进一个 ConcurrentQueue<LogEvent>（内存队列）。

后台单一线程（消费者），从队列里取日志，真正写入磁盘文件。

当消费者线程检测到需要滚动时：它首先暂停从队列取数据，执行 fs.Flush() 和滚动步骤。滚动期间，所有新日志只是在内存 Queue 里堆积，绝不会丢失。等到新文件打开，消费者线程继续消费队列。
