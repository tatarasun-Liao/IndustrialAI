# 第一部分：背景与起源 —— 为什么要发明特性？

**历史背景：**
在 C/C++ 时代，如果我们想给某个函数加一段“描述信息”（比如：这个函数是 API 接口，这个字段在数据库对应某个列），我们通常怎么做？答案是**写宏（Macro）**或者**在头文件里写注释**。但这非常脆弱：宏在预处理阶段就被替换了，注释在编译成二进制后直接消失。

**微软的终极解决思路：**
.NET 需要在编译后的程序集（DLL/EXE）中**持久保存**这些“附加说明”，并且让它在运行时**可被程序自身读取**。于是，CLI（公共语言基础结构）规范引入了**元数据（Metadata）**，而特性（Attribute）就是让开发者向元数据中“塞入自定义结构化数据”的官方桥梁。

# 第二部分：底层原理 —— 特性究竟存在电脑里的什么地方？

如果你用 ILSpy 或 DnSpy 反编译一个带有 `[Obsolete]` 特性的 DLL，你会看到什么？**特性并不执行任何代码，它只是一段二进制序列化的数据。**

### 底层存储结构（重点！）
当我们编译代码 `[MyAttr("Hello", Level=1)] public void Test() {}` 时：
1. **编译器解析**：编译器检查 `MyAttr` 是否为 `Attribute` 的子类，检查传入的参数是否匹配构造函数，检查是否满足 `[AttributeUsage]` 限制。
2. **生成元数据表（Metadata Table）**：编译出的 PE 文件中，有一个 `.custom` 节区（或者称为 `CustomAttribute` 表）。编译器在这里记录了一条记录：
   - **TypeRef**：指向 `MyAttr` 类的元数据 token。
   - **Constructor**：指向 `MyAttr` 的构造函数。
   - **FixedArgs**：位置参数（"Hello"）的二进制序列化值。
   - **NamedArgs**：命名参数（Level=1）的二进制序列化值。
3. **无代码执行**：请注意，**此时 `MyAttr` 的构造函数尚未被调用！** 特性此时仅仅是一段 0101 的二进制 blob 数据，躺在文件里。

# 第三部分：生命周期 —— 从编译到运行，特性经历了什么？

特性的生命周期分为三个阶段，**其中第三个阶段是很多人认知的盲区**。

1. **编译期（Compile-time）**：被编译进元数据（如上述）。（注：C# 12 的 `[StringSyntax]` 甚至只在编译期被 IDE 读取，编译后它可能会被编译器直接剥离，完全不进元数据，这是性能优化）。
2. **加载期（Load-time）**：当 CLR 加载程序集时，并不加载属性。特性是“惰性加载”的。
3. **运行期读取（Runtime Reflection）**：当你调用 `method.GetCustomAttribute<MyAttr>()` 时，会发生这些事：
   - CLR 在元数据表中找到那条 `.custom` 记录。
   - CLR 解析该记录中的二进制数据。
   - CLR 使用 `Activator.CreateInstance` （底层实际上调用了 `RuntimeHelpers` 和构造函数）**实例化 `MyAttr` 对象**（此时才执行构造函数！）。
   - CLR 将解析出的“命名参数”通过反射赋值给实例的属性。
   - 将这个实例返回给你。

> **💡 深度知识点（性能陷阱）：** `GetCustomAttribute` 每次调用都会**重新实例化**一个对象！如果没有缓存，在每秒几万次的调用中，它会频繁触发构造和垃圾回收（GC），导致严重的性能损耗。这就是为什么高级框架（如 ASP.NET Core）会在启动时缓存特性的原因。

# 第四部分：调用逻辑 —— 特性到底是怎么“生效”的？

**一个经典的误区：** 很多人以为贴了 `[Required]`，对象就会自动被验证；贴了 `[HttpGet]`，方法就会自动响应请求。
**底层事实是：特性本身绝不会自动执行任何逻辑！** 特性只是数据（Data-only）。

### 那么逻辑是谁执行的？是“消费者”！

来看底层调用逻辑的结构：

```csharp
// 1. 定义特性（数据）
[AttributeUsage(AttributeTargets.Property)]
public class MustNotBeNullAttribute : Attribute { }

// 2. 标记特性（贴标签）
public class UserEntity {
    [MustNotBeNull]
    public string Name { get; set; }
}

// 3. 编写“消费者”（这是真正触发特性的地方）
public class DataValidator {
    public static bool Validate(object obj) {
        var props = obj.GetType().GetProperties(); // 关键：通过反射获取
        foreach(var p in props) {
            // 关键：查看是否有这个特性
            var attr = p.GetCustomAttribute<MustNotBeNullAttribute>();
            if(attr != null) { 
                // 执行实际逻辑
                if(p.GetValue(obj) == null) return false; 
            }
        }
        return true;
    }
}
```
**结论**：ASP.NET Core 的 `[Required]` 之所以有效，是因为 MVC 框架内部有一个叫 `DataAnnotationsModelValidator` 的“消费者”，它在控制器执行前调用了我们上面写的这种逻辑。AOP（面向切面编程）如 PostSharp 或者 Castle Windsor，也是利用了这个原理，配合 **动态代理（DynamicProxy）** 在运行时拦截方法调用，在调用前后读取特性并执行逻辑。

# 第五部分：深度细节 —— 那些决定生死的参数（Inherited, AllowMultiple）

底层设计极其精密，你必须掌握这两个参数，否则在实际项目中会遇到莫名其妙的 Bug。

### 1. `Inherited` （继承性）
```csharp
[AttributeUsage(AttributeTargets.Class, Inherited = true)]
public class BaseAttribute : Attribute { }
[Base]
public class Parent { }
public class Child : Parent { } // Child 是否拥有 BaseAttribute？
```
- **如果是类（Class）**：`Child` 通过 `GetCustomAttributes(true)` 可以获取到 `BaseAttribute`（因为 `Inherited=true`）。
- **如果是方法/属性（Method/Property）**：底层机制完全不同！如果父类方法 `virtual void Run()` 贴了特性，子类 **`override`** 该方法时，通过反射读取子类方法，**默认将不再包含父类的特性**（除非你在子类方法上重复标记）。这是为了防范子类重写时，特性的语义失效。

### 2. `AllowMultiple` （多标签能力）
某些场景下，我们需要对一个类贴多个相同的特性（如多个权限角色）。
```csharp
[AttributeUsage(AttributeTargets.Method, AllowMultiple = true)]
public class RoleAttribute : Attribute { public string Name {get; set;} }

[Role(Name="Admin")]
[Role(Name="SuperUser")]
public void Manage() { }
```
如果不设为 `true`，编译器直接报错。底层获取时，需要用 `GetCustomAttributes<RoleAttribute>()` 获取一个数组。

# 第六部分：.NET 8 时代的“地震级”变革（AOT 与 源生成器）

在 .NET 8 之前，特性的核心读取方式依赖于 **反射（Reflection）**。但在 .NET 8 全面拥抱 **Native AOT（提前编译）** 后，反射遭遇了致命问题：AOT 编译需要将运行时不需要的代码剪裁掉（Trimmed），而 `GetCustomAttribute` 在编译时**不知道**你会反射哪个属性，于是这些特性代码很容易被剪裁器误删，导致运行时崩溃（`TypeLoadException` 或 `NullReference`）。

**.NET 8 是如何解决这个问题的？**

### 1. 核心护身符：`[DynamicallyAccessedMembers]`
如果要在 AOT 下安全读取特性，你必须在调用反射的方法上标记 `[DynamicallyAccessedMembers(DynamicallyAccessedMemberTypes.PublicMethods)]`。
这告诉 AOT 编译器：“别把这个方法的相关元数据给剪裁掉，我反射的时候要用！”这是底层反射安全的保底方案。

### 2. 终极解决方案：源生成器（Source Generators）
.NET 8 强烈倡导用 **源生成器** 替代运行时反射。
例如 `[GeneratedRegex]`：
```csharp
// 旧世界
if (Regex.IsMatch(email, "^[^@]+@[^@]+$")) { }

// .NET 8 新世界
[GeneratedRegex("^[^@]+@[^@]+$")]
private static partial Regex MyRegex();
```
**底层发生了什么？** 这个特性告诉编译器：“在编译期，帮我生成一段执行正则匹配的 C# 代码”。不再是运行时用反射读取特性并去构建 `Regex` 对象，而是编译期就生成硬编码的代码。**完全零反射、零运行时开销、完美兼容 AOT。**

# 第七部分：实战进阶 —— 手写一个可执行逻辑的“伪 AOP 特性”

为了展示特性的全部威力，我们来手写一个特性，让它看起来像是“它自己执行了逻辑”。实际原理是：**利用 `System.Runtime.CompilerServices` 和静态构造函数，在特性实例化时挂载全局事件钩子。**

```csharp
// 1. 定义一个日志特性
[AttributeUsage(AttributeTargets.Method)]
public class LogExecutionTimeAttribute : Attribute
{
    public void OnEntry() => Console.WriteLine("方法开始执行");
    public void OnExit() => Console.WriteLine("方法执行结束");
}

// 2. 编写一个**框架拦截器**（关键：特性本身不执行，消费者执行）
public static class MethodInterceptor
{
    // 使用委托缓存和反射，模拟框架对特性的"拉取"和"执行"
    public static void ExecuteWithLogging(Action action) 
    {
        var method = action.Method;
        var attr = method.GetCustomAttribute<LogExecutionTimeAttribute>();
        
        if(attr != null)
        {
            attr.OnEntry(); // 真正的逻辑在这里被调用！
            action();
            attr.OnExit();
        }
        else 
        {
            action();
        }
    }
}

// 3. 使用
public class MyBusiness
{
    [LogExecutionTime]
    public static void DoWork() 
    {
        Thread.Sleep(1000);
    }
}
// 调用：
MethodInterceptor.ExecuteWithLogging(MyBusiness.DoWork); 
// 输出：方法开始执行 -> (等待1秒) -> 方法执行结束
```

# 第八部分：总结 —— 特性全景思维导图

至此，你对特性的理解应从“贴标签”上升到“元数据驱动的声明式编程”高度。总结如下：

| 维度 | 详细内容 |
| :--- | :--- |
| **本质** | 嵌入在 PE 文件元数据 `CustomAttribute` 表中的序列化数据流。 |
| **作用** | 为代码附加声明性元数据，实现“告诉框架怎么做，而不是自己手动写判断逻辑”。 |
| **底层传递** | 编译时 -> 二进制位流 -> 运行时延迟实例化 (通过 `Activator`) -> 反射读取。 |
| **真正的调用** | **不直接调用**。由框架（MVC、EF Core、序列化器）通过反射主动拉取数据，触发特定业务逻辑。 |
| **泛型化** | 特性本身不能写 `<T>`。若要支持不同类型，需在特性内部属性中传入 `Type` 类型参数。 |
| **性能铁律** | 避免在高频循环中调用 `GetCustomAttribute`，必须使用 **缓存策略**（如 `ConcurrentDictionary` 缓存反射结果）。 |
| **.NET 8 飞跃** | 反射正走向终结，推荐优先使用 **源生成器（Source Generator）** 和 `[GeneratedXxx]` 来彻底规避运行时反射开销。 |
| **AOT 兼容** | 禁止随意使用 `GetCustomAttribute`，需配合 `[DynamicallyAccessedMembers]` 防止代码裁剪。 |

**再画一个底层视觉化呈现：**
> `[MyAttr(5, Name="A")]` 在底层等于向 CLR 投递了一份这样的订单：
> `{ "Type": "MyAttr", "CtorArg1": 5, "PropArg1": "A", "Target": "MethodA" }`
> 框架收到订单后，才在运行时“打开包裹”组装成对象，并决定是否执行。

如果你能够把上述原理、调用逻辑和 .NET 8 的 AOT 变化理解透彻，你不仅掌握了特性的使用，更彻底弄懂了 .NET 的元数据体系和反射的局限性。如果你在实际项目中遇到了某个特性失效或性能瓶颈，大胆套用上述“反射与缓存”的底层逻辑去排查即可。
