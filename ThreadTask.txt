1. 操作系统层面的进程与线程
进程：资源分配的基本单位，拥有独立的内存空间。

线程：CPU 调度的基本单位，同一进程内的线程共享内存。

2 .NET 1.0：Thread 的蛮荒时代,开发者需要手动管理线程的生命周期、同步和通信

Thread t1 = new Thread(new ThreadStart(Method));
t1.Start();

// 使用 ParameterizedThreadStart 委托（可传一个 object 参数）
Thread t2 = new Thread(ParameterizedMethod);
t2.Start("hello");

// 更简洁的匿名方法 / Lambda
Thread t3 = new Thread(() => { Console.WriteLine("Do work"); });
t3.Start();

// 设置线程属性
t3.IsBackground = true; // 前台线程会阻止进程退出，后台不会
t3.Name = "WorkerThread";
t3.Priority = ThreadPriority.Normal;

· 线程控制（几乎都是坏实践）

t.Join();        // 阻塞当前线程直到 t 结束
t.Abort();       // ⚠️ 已过时，强行终止会引发 ThreadAbortException，破坏状态
Thread.Sleep(0);  // 出让当前时间片
Thread.Sleep(1);  // 让出剩余时间片，至少休眠1ms（取决于系统时钟分辨率）
Thread.Yield();   // 让出处理器给就绪线程

线程锁：
private readonly object _lock = new object();
public void Increment()
{
    lock (_lock)
    {
        _counter++;
    }
}
· Thread 的根本缺陷
创建/销毁成本极高：一个线程默认占用 1MB 栈空间，数千线程会耗尽内存。

资源浪费：大多数时间线程在等待 I/O 或阻塞，却占用着完整的栈和内核对象。

管理复杂：需要手动处理异常、同步、通信，极易出错。

扩展性差：在高并发下，线程数很快成为瓶颈，无法充分利用 CPU。

3.NET 1.1：ThreadPool —— 线程池化
为解决无限制创建线程的问题，CLR 内置了线程池。它会维护少量线程（通常等于核心数），任务放入队列，线程循环获取任务执行。

· 线程池核心机制：队列！！！
全局队列 + 每个工作线程的本地队列。

工作窃取算法：当一个线程的本地队列清空后，它会从其他线程的本地队列尾部窃取任务，避免全局锁竞争。

线程注入与退休：如果所有线程都在忙碌且队列持续增长，池会注入新线程（通常每秒2个），直到达到最大线程数；空闲线程会自行退休。

·ThreadPool 的局限
不支持任务的完成通知、返回值、异常传播。

不适合长期运行的任务：如果线程池线程都被阻塞，后续任务无法执行，导致饥饿。

无法取消或组合。

这促生了 Task 和 async/await。

4 .NET 4.0：Task Parallel Library (TPL)
Task 类代表了“将来会完成的一个异步操作”，是 Task<TResult>（返回结果）和 Task（无返回）的基础。
它统一了线程池工作、I/O 完成、定时器等所有异步模式

·创建与启动
// 方式1：Task.Run (最常用)
Task task = Task.Run(() => DoCpuWork());

// 方式2：Task.Factory.StartNew (更多控制)
Task task = Task.Factory.StartNew(
    () => DoCpuWork(),
    CancellationToken.None,
    TaskCreationOptions.LongRunning,
    TaskScheduler.Default);

 ·Task 的状态与生命周期
Task.Status 枚举：

Created：尚未启动。

WaitingForActivation：尚未被调度（如通过 TaskCompletionSource 创建的）。

WaitingToRun：已在队列中等待执行。

Running：正在执行。

WaitingForChildrenToComplete：等待附加的子任务完成。

RanToCompletion：成功完成。

Faulted：因未处理异常而失败。

Canceled：因 CancellationToken 取消而取消。

·延续（Continuation）
Task<string> original = Task.Run(() => "hello");

original.ContinueWith(t => 
{
    Console.WriteLine(t.Result); // 前一个完成时执行，即使它失败，使用continuewith来拼装task队列
}, TaskContinuationOptions.OnlyOnRanToCompletion);

// async/await 语法糖更优雅：
string result = await original;
Console.WriteLine(result);

·等待：
// 异步等待所有任务完成（不阻塞线程）
await Task.WhenAll(task1, task2, task3);

// 同步等待所有任务完成（阻塞线程）
Task.WaitAll(task1, task2, task3);

// 异步等待任意一个完成
Task<int> firstFinished = await Task.WhenAny(task1, task2);

//同步等待任意一个任务完成
Task.WaitAny(task1, task2);

// 异常处理：WhenAll 失败时用 AggregateException 包装
try
{
    await Task.WhenAll(t1, t2);
}
catch (AggregateException ae)
{
    foreach(var ex in ae.InnerExceptions) ...
}

· Task 与 ThreadPool 的关系
Task.Run 和 Task.Factory.StartNew 默认使用 TaskScheduler.Default，也就是线程池调度器。任务并不直接对应线程：

一个任务可能被分配到任何线程池线程。

异步 await 时，任务可能由一个线程开始，另一个线程继续。

TaskCreationOptions.LongRunning 会提示调度器创建一个专用线程（避免阻塞线程池）。

5 .NET 4.5 与 C# 5.0：async / await
async 和 await 是编译器功能，将复杂的异步回调转化为线性的同步风格代码。其底层依赖 Task 和 “Awaiter” 模式。

· 声明方式
public async Task DoAsync() { await ... }
public async Task<int> CalculateAsync() { return await ...; }
public async ValueTask<int> FastReturnAsync() { return ...; } // 避免堆分配
public async IAsyncEnumerable<int> StreamAsync() { ... }

·await 的作用
await someTask 会检查任务是否已完成；若完成则继续同步执行，否则暂停当前方法，注册一个回调到该任务，然后释放线程回线程池。

编译器将整个方法转换为状态机，把 await 之间的代码划分为多个步骤。

实际效果就是主线程执行到await的时候跳出执行外部调用方法不阻塞，而开了一个新的线程继续执行await内修饰的内容

·异常传播
async 方法内部抛出的异常会被包装在返回的 Task 中。

当 await 该 Task 时，异常被重新抛出（是原始异常，不是 AggregateException）。

未 await 的任务异常会触发 TaskScheduler.UnobservedTaskException 事件。

·取消异步方法
通过传递 CancellationToken 给异步方法，方法内部周期性检查或传给底层 API。

二、CancellationToken 与 CancellationTokenSource取消线程
模式：协作式：请求取消的一方通过 CancellationTokenSource 发出信号，
执行操作的一方通过轮询 CancellationToken 自行决定终止。这避免了 Thread.Abort 的破坏性。

1.声明方式：
// 无超时，手动取消
var cts = new CancellationTokenSource();
CancellationToken token = cts.Token;

// 带超时，自动取消
var cts = new CancellationTokenSource(TimeSpan.FromSeconds(10));

// 链接多个Token，任意一个取消则链接源取消
var linkedCts = CancellationTokenSource.CreateLinkedTokenSource(token1, token2);

// 为 CancellationToken 注册回调
token.Register(() => Console.WriteLine("已被取消"));

2.触发取消:
· 调用方请求取消：
cts.Cancel();       // 请求取消
cts.CancelAfter(5000); // 延迟取消（毫秒）

· 响应方响应取消：
public async Task ProcessDataAsync(CancellationToken token)
{
    while (moreData)
    {
        // 方法1：抛出异常（推荐，符合约定）
        token.ThrowIfCancellationRequested();
        
        // 方法2：判断后返回
        if (token.IsCancellationRequested) return;
        
        // 传递给底层异步API
        await stream.ReadAsync(buffer, token);
    }
}

OperationCanceledException 是取消的标准异常，调用方应捕获它并视为正常终止

三、System.Threading.Channels —— 异步数据管道

Channel<T> 是一种高性能的生产者-消费者异步队列，设计用于在多线程环境中安全地传递数据，并原生支持异步等待和背压控制。
它吸收了 System.Collections.Concurrent 的无锁特性并加入了 TaskCompletionSource 实现高效的异步阻塞。

1.声明方式：
· 无界通道：
Channel<string> unbounded = Channel.CreateUnbounded<string>();
无界通道内部使用无界 ConcurrentQueue<T>，写入永远不会等待，但可能耗尽内存

·有界通道：
var options = new BoundedChannelOptions(1000)
{
    FullMode = BoundedChannelFullMode.Wait   // 写入等待空间，
     SingleWriter = true,  // 优化提示：只有一个生产者，可减少锁
    SingleReader = true   // 只有一个消费者
};
Channel<string> bounded = Channel.CreateBounded<string>(options);

BoundedChannelFullMode 枚举：

Wait：写入异步等待直到有空位（默认）。

DropOldest：丢弃队列中最旧的消息，写入成功。

DropNewest：丢弃正要写入的消息，写入成功。

DropWrite：直接丢弃写入的消息，TryWrite 返回 false，WriteAsync 抛出异常

2.Writer（生产者端）

ChannelWriter<string> writer = channel.Writer;

// 非阻塞写入
if (writer.TryWrite("msg"))
{
    // 成功
}
else
{
    // 失败（有界通道满且模式为 DropWrite 时可能）
}

// 异步写入，可等待，支持取消
await writer.WriteAsync("msg", cancellationToken);

// 标记写入完成，之后不能再写入，Reader读完会停止
writer.Complete();
writer.TryComplete(ex); // 带上异常，Reader端会获取到

3.Reader（消费者端）

ChannelReader<string> reader = channel.Reader;

// 非阻塞读取
if (reader.TryRead(out string msg))
{
    // 处理
}

// 异步读取单条
string msg = await reader.ReadAsync(cancellationToken);

// 等待有数据可读，但不取出
if (await reader.WaitToReadAsync(cancellationToken))
{
    // 此时可确保有数据，再用 TryRead 批量获取
    while (reader.TryRead(out string item))
    {
        // 处理
    }
}

// 异步流式读取 (需要 C# 8.0+)
await foreach (string item in reader.ReadAllAsync(cancellationToken))
{
    // 处理，通道完成或取消时退出循环
}

*************ReadAllAsync 返回 IAsyncEnumerable<T>，
它内部就是反复调用 WaitToReadAsync 和 TryRead，直到通道 Completion 对象完成

4.多生产者多消费者：
Channel 天生支持多生产者写入和多消费者读取，内部所有操作线程安全。消费者们并发调用 ReadAllAsync 将形成自然负载均衡：每条消息只会被一个消费者获取。

var tasks = new Task[consumerCount];
for (int i = 0; i < consumerCount; i++)
{
    tasks[i] = Task.Run(async () =>
    {
        await foreach (var item in reader.ReadAllAsync(ct))
        {
            Process(item);
        }
    });
}
await Task.WhenAll(tasks);

5.背压处理
有界通道结合 BoundedChannelFullMode.Wait 是天然的背压机制：当消费者跟不上时，生产者写入会被异步阻塞，减慢生产速度，
直到消费者腾出空间。

监控通道积压

int pending = reader.Count;

6.完成与异常处理
写入端调用 writer.Complete() 后，reader 端在读完剩余所有数据后，ReadAllAsync 循环自然退出。

如果在调用 Complete 时传入异常（ChannelClosedException），那么 reader 端 await foreach 会抛出该异常。

消费者也可通过 reader.Completion 属性获知通道已关闭：

await reader.Completion; // 当 Writer 调用 Complete() 后此 Task 完成

7.源码级理解：内部如何实现异步等待
无界通道内部主要由 ConcurrentQueue<T> 和 Deque<AsyncOperation<T>> 实现。

当 ReadAsync 发现队列为空，会创建一个 AsyncOperation<T> 结构（包含一个 ValueTask<T> 的源）并加入等待者队列，然后返回该 ValueTask。

当 WriteAsync 写入数据后，检查是否有等待的读者，有则直接将数据通过 TrySetResult 设置给那个异步操作，从而唤醒正在 await 的消费者，整个过程无锁无竞争，极快。

有界通道则多一个信号量（SemaphoreSlim）控制容量，配合写等待队列。

8. 性能建议
如果场景是高吞吐、低延迟，尽量使用 单消费者/单生产者 选项减少锁开销。

批量写入或批量读取可以减少异步调用次数。

对于不需要异步等待的场景（如内存内流水线，数据总是能立刻处理），ConcurrentQueue 仍更简单高效

四、 IAsyncEnumerable<T> 与 await foreach
IAsyncEnumerable<T> 以支持异步拉取数据流，弥补了之前只能一次性返回整个 Task<IEnumerable<T>> 的不足。非常适合数据库游标、分页 API、事件流等

1.声明方法：
public async IAsyncEnumerable<int> GenerateNumbersAsync(
    [EnumeratorCancellation] CancellationToken ct = default)
{
    for (int i = 0; i < 10; i++)
    {
        ct.ThrowIfCancellationRequested();
        await Task.Delay(100, ct); // 异步工作
        yield return i;            // 每次 yield 控制权返回调用者
    }
}

注：yield关键字与IEnumerable迭代器组合使用，只有在进行遍历时才有效果，如果不添加yield关键字，则遍历需要完全结束后才能返回结果，且占用完整遍历结果空间，
但是加上yield后会形成迭代器模式，实现遍历一个数据，返回一个数据，并且每次遍历只占用一个元素的堆栈内存，适合及大量数据

实现原理：
编译器将 async 迭代器方法转换为一个实现了 IAsyncEnumerable<T> 和 IAsyncEnumerator<T> 的状态机。每次调用 MoveNextAsync() 都会执行一段状态机的代码直到下一个 yield return 或结束。

2.消费异步迭代器

//由于GenerateNumbersAsync内部使用yield实现了迭代器模式，在外部调用时可以实现外面循环一次，找GenerateNumbersAsync方法要一个最新的遍历数据
而不是等待GenerateNumbersAsync返回完整的列表后再foreach循环，可以节约内存，增加吞吐量
await foreach (int number in GenerateNumbersAsync().WithCancellation(cts.Token))
{
    Console.WriteLine(number);
}
// 循环结束后，异步枚举器会被 Dispose

3.使用 System.Linq.Async 包：提供 Where, Select, Take, ToListAsync 等 LINQ 操作符。

var result = await GenerateNumbersAsync()
    .Where(x => x % 2 == 0)
    .Select(x => x * 2)
    .ToListAsync(cts.Token);
